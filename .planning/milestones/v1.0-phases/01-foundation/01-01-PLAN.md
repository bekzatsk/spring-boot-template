**Kotlin compiler plugins** — update the existing `kotlin-maven-plugin` configuration:
- Add `<plugin>jpa</plugin>` and `<plugin>all-open</plugin>` to `<compilerPlugins>` (alongside existing `spring` plugin)
- Add `<pluginOptions>` block with all-open annotations for `jakarta.persistence.Entity`, `jakarta.persistence.MappedSuperclass`, `jakarta.persistence.Embeddable`
- Add `kotlin-maven-noarg` dependency to plugin dependencies (alongside existing `kotlin-maven-allopen`)

**TemplateApplication.kt** — add exclusion to prevent Spring Authorization Server auto-configuration from activating a full OAuth2 authorization server:
```kotlin
@SpringBootApplication(exclude = [
    org.springframework.boot.autoconfigure.security.oauth2.server.servlet.OAuth2AuthorizationServerAutoConfiguration::class
])
```
This prevents the auto-config from creating a competing SecurityFilterChain. We only need the JWT infrastructure beans (JWKSource, NimbusJwtEncoder), NOT the full authorization server.

Note: The exact auto-configuration class name may differ in Spring Boot 4.0.3. If the class above does not resolve, check the Spring Boot 4 autoconfiguration package. The Spring Security 7 merged authorization server may use a different class path — look for the autoconfiguration class in the `spring-boot-autoconfigure` JAR that registers the authorization server SecurityFilterChain.

**Verify the dependency resolves:** Run `./mvnw dependency:resolve` to confirm all artifacts download correctly. If `spring-boot-starter-oauth2-authorization-server` fails, try `spring-boot-starter-security-oauth2-authorization-server` instead (Spring Boot 4.0.3 may have already renamed it).
  </action>

If datasource auto-config prevents compile, temporarily verify with `./mvnw dependency:resolve -q` instead and note that datasource config comes in the next task.

**docker-compose.yml** — create at project root:
```yaml
services:
  postgres:
    image: postgres:18
    container_name: template-postgres
    environment:
      POSTGRES_DB: ${DB_NAME:-template}
      POSTGRES_USER: ${DB_USERNAME:-template}
      POSTGRES_PASSWORD: ${DB_PASSWORD:-template}
      PGDATA: /var/lib/postgresql/data
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U template"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  postgres_data:
```
Do NOT include `version: '3.9'` — modern Docker Compose does not need it.

**application.yaml** — replace existing content with multi-document YAML:

Common section (no profile activation):
- `spring.application.name: template`
- `spring.threads.virtual.enabled: true` (INFR-01)
- `spring.jpa.open-in-view: false` (stateless API best practice)
- `server.port: 7070`

Dev profile section (separated by `---`):
- `spring.config.activate.on-profile: dev`
- `spring.datasource.url: jdbc:postgresql://localhost:5432/${DB_NAME:template}`
- `spring.datasource.username: ${DB_USERNAME:template}`
- `spring.datasource.password: ${DB_PASSWORD:template}`
- `spring.jpa.hibernate.ddl-auto: create-drop`
- `spring.jpa.show-sql: true`
- `spring.jpa.properties.hibernate.format_sql: true`
- No keystore config (dev uses auto-generated in-memory RSA keypair)

Prod profile section (separated by `---`):
- `spring.config.activate.on-profile: prod`
- `spring.datasource.url: ${DATABASE_URL}`
- `spring.datasource.username: ${DB_USERNAME}`
- `spring.datasource.password: ${DB_PASSWORD}`
- `spring.jpa.hibernate.ddl-auto: validate`
- `app.security.jwt.keystore-location: ${JWT_KEYSTORE_LOCATION}`
- `app.security.jwt.keystore-password: ${JWT_KEYSTORE_PASSWORD}`
- `app.security.jwt.key-alias: ${JWT_KEY_ALIAS:jwt}`

**IMPORTANT for SECU-09:** Dev profile defaults (`:template`) are acceptable for local development convenience. Prod profile MUST NOT have any defaults for sensitive values — bare `${ENV_VAR}` only.

Also add to the common section to make Spring Security permissive by default during Phase 1 (no endpoints to protect yet):
- `spring.security.user.name: ${SPRING_SECURITY_USER:admin}`
- `spring.security.user.password: ${SPRING_SECURITY_PASSWORD:admin}`
Note: This is temporary — Phase 2 replaces it with a full SecurityFilterChain. It prevents Spring Security's default form login from interfering during Phase 1 development.
