**UserService.kt -- rewrite all three findOrCreate* methods:**

1. `findOrCreateGoogleUser(providerId, email, name, picture)`:
   - Lookup: `userRepository.findByEmail(email)`
   - If found: link GOOGLE to existing account:
     - `existing.providers.add(AuthProvider.GOOGLE)` (idempotent)
     - `existing.providerIds[AuthProvider.GOOGLE] = providerId` (update sub on every login)
     - Only update name/picture if currently null on existing user (do NOT overwrite)
     - Return `userRepository.save(existing)`
   - If not found: create new user with `User(email = email).also { it.providers.add(AuthProvider.GOOGLE); it.providerIds[AuthProvider.GOOGLE] = providerId; it.name = name; it.picture = picture }`
   - Keep `@Transactional`

2. `findOrCreateAppleUser(providerId, email, name)`:
   - Step 1: Try `userRepository.findByAppleProviderId(providerId)` -- covers returning users where email is absent. If found, return immediately (no changes needed).
   - Step 2: If not found by Apple sub, email MUST be present (first sign-in). Throw `BadCredentialsException("Email not present in Apple identity token on first sign-in")` if email is null.
   - Step 3: `userRepository.findByEmail(resolvedEmail)` -- link APPLE to existing account if found:
     - `existing.providers.add(AuthProvider.APPLE)`
     - `existing.providerIds[AuthProvider.APPLE] = providerId`
     - Only update name if currently null
     - Return `userRepository.save(existing)`
   - Step 4: Create new user if no match: `User(email = resolvedEmail).also { it.providers.add(AuthProvider.APPLE); it.providerIds[AuthProvider.APPLE] = providerId; it.name = name }`
   - Keep `@Transactional`

3. `findOrCreatePhoneUser(phoneE164)`:
   - Lookup: `userRepository.findByPhone(phoneE164)` -- wait, the repository does not have findByPhone. Use `userRepository.findByEmail("")` which would return ANY phone user (wrong). Instead, keep phone lookup by checking the phone field. Add `fun findByPhone(phone: String): User?` to UserRepository. Actually -- the research says phone users have `email = ""` so findByEmail("") would match all phone users. The correct approach: add `fun findByPhone(phone: String): User?` to UserRepository (Spring Data derived query on the `phone` column which is unique).
   - UPDATE: Also add `fun findByPhone(phone: String): User?` to UserRepository.kt (this is a small addition to Plan 01's file, but since Plan 01 won't be re-executed, add it here in this task).
   - If found: return existing user
   - If not found: create `User(email = "").also { it.providers.add(AuthProvider.LOCAL); it.phone = phoneE164 }`
   - Keep `@Transactional`

**IMPORTANT: Add findByPhone to UserRepository.kt** -- this was not in Plan 01 but is needed for phone user lookup after removing findByProviderAndProviderId. Add `fun findByPhone(phone: String): User?` to UserRepository.kt in this task.

4. `findById(id)`: No changes needed.

**LocalAuthService.kt -- rewrite register() and login():**

1. `register(email, rawPassword, name)`:
   - Lookup: `userRepository.findByEmail(email)`
   - If found AND `passwordHash != null`: throw `IllegalStateException("Email already registered")` (409 Conflict, same as before)
   - If found AND `passwordHash == null`: this is an existing Google/Apple user adding LOCAL credentials:
     - `existing.providers.add(AuthProvider.LOCAL)`
     - `existing.passwordHash = passwordEncoder.encode(rawPassword)`
     - Only update name if currently null
     - `userRepository.save(existing)`, then issue tokens
   - If not found: create new user with `User(email = email).also { it.providers.add(AuthProvider.LOCAL); it.passwordHash = passwordEncoder.encode(rawPassword); it.name = name }`, then issue tokens
   - Keep `@Transactional`

2. `login(email, rawPassword)`:
   - First authenticate via `authenticationManager.authenticate(UsernamePasswordAuthenticationToken.unauthenticated(email, rawPassword))` -- this calls LocalUserDetailsService
   - Then: `userRepository.findByEmail(email) ?: throw BadCredentialsException("User not found")`
   - Issue tokens
   - No mutation -- login is read-only (no @Transactional needed, but harmless to keep)

**LocalUserDetailsService.kt -- rewrite loadUserByUsername():**

1. Lookup: `userRepository.findByEmail(email) ?: throw UsernameNotFoundException("No account for email: $email")`
2. Check: `if (AuthProvider.LOCAL !in user.providers)` -- throw `UsernameNotFoundException("No local account for email: $email")` (user exists but registered via Google/Apple only, no local credentials)
3. Check: `if (user.passwordHash == null)` -- throw `BadCredentialsException("No password set")`
4. Build and return `UserDetails` with the user's passwordHash and roles (same pattern as before)
